## Vulnerability Overview

CVE-2023-27372 is an unauthenticated remote code execution vulnerability affecting SPIP versions prior to the patches released in March 2023. The vulnerability stems from improper handling of deserialization in the `oubli` parameter of the password reset form.

<Warning>
  All SPIP installations running versions below the patched releases are vulnerable to unauthenticated remote code execution. Immediate upgrade is strongly recommended.
</Warning>

## Version Check

To determine your SPIP version, check the file `ecrire/inc_version.php` or look at the footer of your SPIP administration panel.

<Info>
  If you're running any version older than those listed above, your installation is vulnerable and should be upgraded immediately.
</Info>

## Remediation

### Immediate Actions

1. **Identify your current SPIP version**
   - Check `ecrire/inc_version.php`
   - Or view the admin panel footer

2. **Upgrade to a patched version**
   - 3.2.x users → upgrade to 3.2.18 or later
   - 4.0.x users → upgrade to 4.0.10 or later
   - 4.1.x users → upgrade to 4.1.8 or later
   - 4.2.x users → upgrade to 4.2.1 or later

3. **Verify the upgrade**
   - Confirm the new version in the admin panel
   - Test password reset functionality

4. **Monitor for exploitation attempts**
   - Review web server logs for POST requests to `/spip.php?page=spip_pass`
   - Look for suspicious serialized data in the `oubli` parameter
   - Check for unusual process execution by the web server user

<Warning>
  If you suspect your server has been compromised, perform a full security audit, review all system logs, check for unauthorized user accounts, and consider restoring from a clean backup.
</Warning>

## Detection

Look for the following indicators in your web server logs:

```
POST /spip.php?page=spip_pass
```

With POST data containing serialized PHP payloads in the `oubli` parameter:

```
oubli=s:XX:"<?php system('...');
```

<Info>
  The exploit requires the anti-CSRF token to function, so attackers will typically make a GET request first to retrieve the token, followed by a POST request with the malicious payload.
</Info>
